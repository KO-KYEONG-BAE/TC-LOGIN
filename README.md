# SauceDemo 로그인 자동화 테스트
---
단순 기능 테스트를 넘어, 실제 사용자 흐름과 보안 입력까지 고려한 로그인 자동화 테스트 프로젝트
---
## 프로젝트 소개

SauceDemo 사이트의 로그인 기능을 대상으로
Selenium과 Pytest를 사용하여 자동화 테스트를 작성했습니다.

처음에는 로그인 성공/실패만 확인하는 수준으로 시작했지만,
진행하면서 다양한 예외 상황과 보안 입력까지 테스트 범위를 확장했습니다.

---

## 테스트 목적

* 로그인 기능이 정상적으로 동작하는지 확인했습니다
* 잘못된 입력에 대해 적절한 에러 메시지가 출력되는지 검증했습니다
* 비정상 입력(XSS, SQL Injection)에 대한 대응을 확인했습니다
* 로그인 이후 상태 및 접근 제어(로그아웃, 뒤로가기 등)를 검증했습니다

---

## 테스트 설계 근거

로그인 기능은 서비스의 핵심 진입점이므로,
단순 성공/실패뿐 아니라 다음을 중점적으로 검증했습니다

- 인증 실패 시 시스템의 일관된 에러 처리
- 로그인 이후 세션 상태 유지 여부
- 비정상 입력에 대한 보안 취약성 여부

## 사용 기술

* Python 3.11
* Selenium
* Pytest
* Chrome WebDriver

---

## 테스트 케이스

### 로그인 관련

* 정상 로그인 테스트를 수행했습니다
* 잘못된 비밀번호 입력을 테스트했습니다
* 존재하지 않는 계정으로 로그인 시도를 테스트했습니다
* username 미입력 상태를 테스트했습니다
* password 미입력 상태를 테스트했습니다
* username/password 모두 미입력 상태를 테스트했습니다
* 비정상 형식의 username 입력을 테스트했습니다
* locked 계정 로그인 시도를 테스트했습니다

---

### 보안 테스트

* SQL Injection 입력(`'OR '1'='1`)을 테스트했습니다
* XSS 입력(`<script>alert(1)</script>`)을 테스트했습니다

---

### 상태 / 흐름 테스트

* 로그아웃 이후 inventory 페이지 접근 차단 여부를 테스트했습니다
* 로그아웃 이후 뒤로가기 시 접근 가능 여부를 테스트했습니다

---

### 성능 테스트

* 로그인 응답 시간을 측정했습니다 (5초 기준)

---

## 구현하면서 신경 쓴 부분

### 1. parametrize 사용

여러 입력 케이스를 parametrize로 통합하여
테스트 코드의 중복을 제거하고 확장성을 확보했습니다.

---

### 2. fixture 사용

WebDriver 생성 및 종료를 fixture로 분리하여
테스트 코드의 반복을 줄였습니다.

---

### 3. explicit wait 사용

초기에는 find_element를 바로 사용했지만,
요소 로딩 타이밍 문제로 인해 테스트가 불안정했습니다.

이를 해결하기 위해 WebDriverWait을 사용하여
요소가 존재하거나 클릭 가능한 상태가 될 때까지 기다리도록 수정했습니다.

---

### 4. 상태 기반 테스트

단순 로그인 기능 검증에서 끝나지 않고
로그아웃 이후 접근, 뒤로가기 등 실제 사용자 흐름을 고려하여 테스트를 확장했습니다.

## 테스트 기준

* 로그인 성공: inventory 리스트가 표시되는지 확인했습니다
* 로그인 실패: 에러 메시지 텍스트를 검증했습니다
* 응답 시간: 5초 이하 기준으로 설정했습니다

응답 시간은 Selenium 기반으로 브라우저 렌더링 시간을 포함한
체감 성능 기준으로 참고용으로 사용했습니다.

---

## 느낀 점

단순히 자동화를 구현하는 것에서 끝나는 것이 아니라,
어떤 상황을 테스트해야 하는지를 고민하는 과정이 중요하다고 느꼈습니다.

특히 로그인 이후의 상태나 예외 케이스를 고려하는 것이
테스트 품질에 큰 영향을 준다고 생각했습니다.
